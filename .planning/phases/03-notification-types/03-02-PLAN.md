# Plan 03-02: Toast Display Integration — Hero Image & Sound

## Objective

Extend the Toast display system to support per-type hero images and sounds. Wire the classification logic from 03-01 into the hook handlers so each notification type triggers a visually distinct Toast with appropriate audio.

## Tasks

<task id="1">
<title>Extend show_toast to support hero image and audio parameters</title>
<read_first>
- src/toast.rs (current show_toast implementation with XML template)
- src/notification.rs (NotificationType enum and its methods)
- .planning/phases/03-notification-types/03-RESEARCH.md §1 and §2 (Toast XML hero image and audio syntax)
</read_first>
<action>
Modify `src/toast.rs`:

1. Add new public function `show_typed_toast(ntype: NotificationType, body: &str, attribution: &str) -> Result<(), AppError>`:
   - Call `crate::assets::ensure_hero_image(ntype)` to get hero image path (Option<PathBuf>)
   - Get `ntype.audio_src()` for the audio element
   - Get `ntype.title()` for the title text
   - Build Toast XML using the new template (see below)
   - If hero image path is Some, include `<image placement="hero" src="file:///{path}"/>` (convert backslashes to forward slashes for the file:/// URI)
   - If hero image path is None, omit the `<image>` element entirely (graceful degradation)
   - For Error type, add `loop="false"` attribute to audio element
   - Create and show toast notification (same WinRT API calls as existing show_toast)

2. New XML template for `show_typed_toast`:
```xml
<toast>
  <visual>
    <binding template="ToastGeneric">
      <text>{title}</text>
      <text>{body}</text>
      <text placement="attribution">{attribution}</text>
      {hero_image_element}
    </binding>
  </visual>
  <audio src="{audio_src}"{loop_attr}/>
</toast>
```

Where:
- `{hero_image_element}` is either `<image placement="hero" src="file:///{path}"/>` or empty string
- `{loop_attr}` is ` loop="false"` for Error type, empty string otherwise
- All text values pass through `escape_xml()` (existing function)
- Hero image path uses forward slashes: `path.to_string_lossy().replace('\\', "/")`

3. Keep existing `show_toast()` function intact (used by nothing after this phase, but avoid breaking changes during development).

4. Add `use crate::notification::NotificationType;` and `use crate::assets;` imports.
</action>

- src/toast.rs contains `placement="hero"`
- src/toast.rs contains `ensure_hero_image`

- src/toast.rs contains `loop="false"`

<action>
Modify `src/hook.rs`:

1. Add import: `use crate::notification::{classify_stop, classify_notification, body_text};`

2. Rewrite `handle_stop(input: &HookInput)`:
```rust
fn handle_stop(input: &HookInput) -> Result<(), crate::error::AppError> {
    if input.stop_hook_active == Some(true) {
        return Ok(());
    }

    let ntype = classify_stop(input);
    let project_name = extract_project_name(&input.cwd);
    let body = body_text(ntype, input);

    crate::toast::show_typed_toast(ntype, &body, &project_name)
}
```

3. Rewrite `handle_notification(input: &HookInput)`:
```rust
fn handle_notification(input: &HookInput) -> Result<(), crate::error::AppError> {
    let ntype = classify_notification(input);
    let project_name = extract_project_name(&input.cwd);
    let body = body_text(ntype, input);

    crate::toast::show_typed_toast(ntype, &body, &project_name)
}
```

4. Extract project name helper (DRY refactor):
```rust
fn extract_project_name(cwd: &str) -> String {
    std::path::Path::new(cwd)
        .file_name()
        .unwrap_or_default()
        .to_string_lossy()
        .to_string()
}
```

5. Keep existing tests in hook.rs unchanged (they test parsing, not display).
</action>
