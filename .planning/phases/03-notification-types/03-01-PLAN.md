# Plan 03-01: Notification Type Classification & Assets Module

## Objective

Create the `NotificationType` enum with full classification logic for all 4 notification types, plus the asset embedding/extraction module for hero images. This is the foundation that the toast display plan (03-02) builds upon.

## Tasks

<task id="1">
<title>Create NotificationType enum and classification functions</title>
<read_first>
- src/hook.rs (current HookInput struct, handle_stop/handle_notification logic)
- .planning/phases/03-notification-types/03-CONTEXT.md (decisions D-01 through D-19)
</read_first>
<action>
Create `src/notification.rs` with:

1. `NotificationType` enum with variants: `TaskComplete`, `PermissionRequest`, `Question`, `Error`

2. Impl block on `NotificationType` with methods:
   - `title(&self) -> &'static str` returning:
     - TaskComplete: `"✔ Task Complete"`
     - PermissionRequest: `"🔐 Permission Required"`
     - Question: `"❓ Question"`
     - Error: `"⚠ Error"`
   - `audio_src(&self) -> &'static str` returning:
     - TaskComplete: `"ms-winsoundevent:Notification.Default"`
     - PermissionRequest: `"ms-winsoundevent:Notification.Reminder"`
     - Question: `"ms-winsoundevent:Notification.IM"`
     - Error: `"ms-winsoundevent:Notification.Looping.Alarm"`
   - `hero_filename(&self) -> &'static str` returning:
     - TaskComplete: `"hero-task-complete.png"`
     - PermissionRequest: `"hero-permission.png"`
     - Question: `"hero-question.png"`
     - Error: `"hero-error.png"`
   - `audio_loop(&self) -> bool` returning `false` for all (Error uses Looping.Alarm but single-play)

3. `classify_stop(input: &HookInput) -> NotificationType` function:
   - Priority order: Error > Question > TaskComplete (D-02)
   - Error detection (D-03, D-14): check `last_assistant_message` lowercase against ERROR_PATTERNS
   - Question detection (D-04, D-15): check `last_assistant_message.trim().ends_with('?')`
   - Default (D-16): `TaskComplete`

4. `classify_notification(input: &HookInput) -> NotificationType` function:
   - D-17: `notification_type == Some("permission_prompt")` → PermissionRequest
   - D-19: fallback when notification_type is None — check `message` contains "permission" (case-insensitive)
   - D-18: any other notification_type → Question

5. `const ERROR_PATTERNS: &[&str]` with values:
   `["api rate limit", "rate limit exceeded", "session limit", "context window", "api error", "authentication failed"]`

6. `pub fn body_text(ntype: NotificationType, input: &HookInput) -> String` function:
   - TaskComplete: `"✔ Task Complete"` (same as title for now)
   - PermissionRequest: use `input.message` or `"Permission needed"`
   - Question: use last line of `input.last_assistant_message` (truncated to 200 chars) or `"Claude has a question"`
   - Error: use relevant portion of `input.last_assistant_message` (truncated to 200 chars) or `"An error occurred"`
</action>

1. Four `include_bytes!()` constants:
   ```rust
   const HERO_TASK_COMPLETE: &[u8] = include_bytes!("../assets/hero-task-complete.png");
   const HERO_PERMISSION: &[u8] = include_bytes!("../assets/hero-permission.png");
   const HERO_QUESTION: &[u8] = include_bytes!("../assets/hero-question.png");
   const HERO_ERROR: &[u8] = include_bytes!("../assets/hero-error.png");
   ```

2. `pub fn assets_dir() -> PathBuf` — returns `%LOCALAPPDATA%\claude-win-notify\assets`
   (mirrors pattern from `log.rs` `log_dir()`)

3. `pub fn ensure_hero_image(ntype: NotificationType) -> Option<PathBuf>` function:
   - Get assets_dir()
   - Build path: `assets_dir().join(ntype.hero_filename())`
   - If file exists, return `Some(path)`
   - If not, `create_dir_all` then `fs::write(path, data)` where data is matched from the `include_bytes!` constants
   - On any IO error: log warning via `crate::log::log_error()`, return `None` (graceful degradation per D-09)
   - Return `Some(path)` on success

4. Helper function `fn hero_data(ntype: NotificationType) -> &'static [u8]` that matches the enum variant to the correct `include_bytes!` constant.
</action>
